## 발견 사항 (우선순위순)

### F-1 · `--adults` 기본값 2의 함정 (혼자 여행 오염) — ✅ 수정 반영 (2026-07-07)
- 증상: 스페인(혼자) 런에서 검색·`getStayDetail` 체이닝 모두 성인 2로 실행, 예약 URL에 `adultCount=2` 고정.
- 원인: `core.cli scenario`의 `--adults` 기본값 2(`core/cli.py:377`). `--companions`는 자유 텍스트라 인원과 연결되지 않고, `commands/scenario.md`도 `--adults` 지정을 지시하지 않음. 전파 자체(`planning.py:98,121`)는 일관됨 — **입구의 침묵 기본값** 문제.
- 반영 내용: scenario.md 실행 예시에 `--adults` 필수 표기 + 동행자→인원 매핑 지시. CLI는 `--adults` 미지정 시 `DEFAULT_ADULTS`(2)로 동작하되 `_meta.warnings`에 경고를 남김(회귀 테스트 2건 추가).

### F-2 · 숙소 검색가 vs 상세가 2.0× 불일치 (가격 의미론) — ✅ 종결: 재현 불가, 인원 종속 가격으로 판정 (2026-07-09)
- 증상: 그라나다 게스트하우스 — 검색 73,739원/박 vs 상세 147,478원/박(**정확히 2배**). 오사카 APA 호텔은 두 값 일치.
- 검증(2026-07-09): 민박·호스텔 3건(까사보니따 5008273·토크 호스텔 1471684·라스트 사무라이 2027325) × adults 1/2 = 5회 비교 → **전건 검색가≈상세가**(차이 0~0.7%, 세금/캐시 수준). 2× 불일치 재현 불가.
- 판정: 이 유형은 **가격이 `adultCount`에 비례**(인원 과금 — 호텔은 객실 단위라 무관, APA가 일치했던 이유). adults=1 검색가 73,438원이 당시 검색가 73,739원과 사실상 동일 → 당시 2×는 "1인 가격 vs 2인 가격", 즉 검색·상세 간 인원 파라미터 불일치(F-1 계열) 또는 이후 수정된 업스트림 표시 문제. 어댑터 수정 불필요.
- 반영: `field_notes.md` §3 가격 의미론 신설, `scenario.md` §출력-2에 "숙소 가격 인용 시 조회 인원 명시" 주의 추가.

### F-3 · 국제선 일부 응답에서 정규화 가격 미반환 — ✅ 종결: 재현 불가, 오독 추정 (2026-07-07)
- 조사: BCN 왕복(raw 2회, 오케스트레이션 인자 포함)·DAD 왕복(오케스트레이션 전체 경로) 재현 → **전건 `price.amount` 정상 반환** (BCN 675,700원~/왕복 1,246,700원, DAD 373,000원~).
- 판정: 업스트림 일시 현상이 아니면, 항공은 `price.text`가 항상 null이고 값이 `amount`에 있는 Price 형태를 실행 세션이 오독("가격 텍스트가 노출되지 않음"이라는 스페인 런 표현과 부합). 어댑터 수정 불필요.
- 반영: scenario.md §출력-2에 "가격 읽는 법"(amount vs text) 명시.

### F-4 · TNA 제목↔gid 페어링 한 칸 밀림 — ✅ 원인 확정·수정 (2026-07-07)
- 재현: `getTnaDetail(3881289)` = "야간 공항↔호이안 픽업" — 다낭 검색 결과에서 3881289를 "원데이 투어"로 표기했던 것과 불일치. **상세가 아니라 `searchTnas` 정규화가 범인.**
- 원인: `_tnas`가 "N번째 distinct `products/` URL → N번째 copy_text 제목"으로 짝지었는데, 다낭 첫 카드가 구형 `offers/63983` 링크(정규식 미포착)라 **전체 gid가 한 칸씩 밀림**.
- 수정: widget 내 **제목 위치 기준으로 뒤따르는 상품 URL을 짝짓기**(제목 미발견 시 gid None — 틀린 짝보다 안전) + `offers/` 패턴 인식. 교차 검증: `getTnaDetail(3520244)` 포함사항이 원데이 투어와 일치.
- 회귀 고정: `fixtures/tnas/searchTnas.sample3.json` + 골든 2건(`TnaPairingGolden`). 기존 오사카 fixture 페어링은 변화 없음(선행 offers 카드 없는 경우 두 방식이 일치).
- 부수 발견: getTnaDetail copy_text 첫 굵은 텍스트가 상품명이 아닌 **옵션명**인 상품 존재(3520244 "아동, 1인 금액") → field_notes §4에 주의 추가. 다낭 스모크 표의 "바나힐&호이안 → 3881289" 표기는 이 버그의 산출물이므로 **해당 시나리오의 TNA gid 인용은 무효**.

### F-5 · 미팅 포인트/가이드 연락처 필드 — ✅ 조사·부분 수정 (2026-07-07)
- 조사 결과(6개 상품 실측): 원래 가설("미팅 필드가 데이터 계층에 아예 없다")은 **부분 오류였다**.
  - **미팅 장소/시간 = 소스에 있음.** `getTnaDetail` **widget "이용 안내" 섹션**에 `장소:`/`시간:` Text 노드로 6개 중 5개 일관 존재(copy_text엔 없어 어댑터가 그동안 못 뽑음).
  - **가이드 긴급 연락처 = 예약 전 소스에 없음.** 전부 "예약 확정 후 조율/안내" — 정규화할 소스 필드 자체가 없음.
- 수정: `ProductCandidate`에 `meeting_place`/`meeting_time` 추가, `_tna_detail`에 widget 워커(`_extract_meeting`)로 정규화. 골든 1건(`TnaMeetingGolden`), 라이브 3상품 검증. 가이드 연락처는 의도적으로 미추출(부재 명시).
- 실험 함의 갱신: 스페인 런의 "가이드 긴급 연락처 자동 노출" 실험 중 **미팅 정보 재확인 카드(T-24h)는 이제 데이터로 뒷받침 가능**. 반면 **긴급 연락처 노출은 여전히 파트너 입력 스키마 정비가 0단계**(voc-025의 일반=긴급 동일 등록 문제 포함) — 상품 응답이 아니라 파트너 온보딩 계층의 과제로 확정.
