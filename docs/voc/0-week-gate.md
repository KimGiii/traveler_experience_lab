## 수집 방법 & 비식별 규칙

1. 공개 웹 검색(마찰 유형 6종 × 상품 도메인별 쿼리)으로 소스 확보.
2. **원문을 저장하지 않고** 마찰 구조만 남긴 요약문으로 재작성.
3. 고객명·전화·이메일·예약번호·결제정보·개인 식별 가능 서술은 요약 단계에서 제거 (수록 금액은 언론 보도의 공개 수치만 허용).
4. 각 레코드에 `evidence_grade` 부여:
   - `direct` — MRT 특정 사용자 사례/언론 보도 (14건)
   - `policy` — 공식 문서(도움말/파트너 안내)가 마찰 구조를 입증 (18건)
   - `industry` — 산업 일반, MRT 특정 아님 (1건, diagnose 시 참고용으로만)

`diagnose.md`의 분류 축(여정 단계/마찰 유형/심각도×빈도)과 레코드 필드가 1:1 대응하도록 설계됨.

## 한계 (diagnose 출력 해석 시 명시할 것)

- **표본 편향**: 불만 중심 검색으로 수집 → 만족 사례 부재. 마찰의 "존재" 근거이지 "비율" 근거가 아님.
- **빈도 추정 불가**: `frequency_hint`는 정성 추정. 실제 발생량은 사내 CS 볼륨 데이터 없이는 알 수 없음.
- **severity는 수집자 추정**: 실측 아님.
- `policy` 레코드는 실제 불만 접수가 아니라 "마찰을 유발하는 정책 구조"의 존재 증명임.

## 사내 경로 재개 조건 (사람 액션)

1. CS/리뷰/커뮤니티 요약 데이터 오너 확인 및 접근권 요청.
2. 비식별 요약 30~50건 export 가능 여부 확인 (원문·PII 반입 금지 원칙 유지).
3. 확보 시 본 fixture를 사내 데이터 기반으로 교체하고, 이 문서의 판정을 ✅로 갱신.
